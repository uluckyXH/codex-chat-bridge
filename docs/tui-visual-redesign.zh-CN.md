# TUI 视觉重设计方案

## 目标

在保持所有按键逻辑、页面跳转、业务动作不变的前提下，对 Chat Codex Ink TUI 的视觉表现做系统性重设计。

核心诉求：

- 全屏占满终端，不再显示为固定宽度小框。
- 橙黄色系作为主配色，搭配协调辅助色。
- 信息分区边界清晰、明显，分区标题醒目。
- 不同交互界面各自有专属视觉风格，避免千篇一律。

不在范围：

- 不修改任何按键绑定或快捷键逻辑。
- 不修改页面跳转规则。
- 不修改 actions/services 层业务逻辑。
- 不修改 `routes.json`、`session-owners.json` 等数据结构。

## 开发规范对齐

本方案遵守 `development-and-test.zh-CN.md` 的以下要求：

- 改动严格限于 TUI 展示层文件（`ui-components.tsx`、`views.tsx`、`runtime-log.tsx`、`run-tui.tsx`、`run-runtime-log.tsx`），不跨层依赖。
- 不在组件里引入新的业务分支或渠道特例。
- 每个文件行数保持在合理范围，不堆砌；颜色常量集中到 `THEME` 对象，不散落各处。
- 实现完成后需有中文测试报告，放入 `reports/tests/`。

### 日志内容不截断原则

运行期日志（`RuntimeLogView`）的 **消息正文**（`entry.message`）必须完整输出，不允许调用 `truncate()` 裁剪：

- 消息文本按终端宽度自动换行（`wrap="wrap"`），不人为截断。
- 多行消息逐行展示，每行前缀 2 空格缩进。
- 日志头部元信息（时间戳、类型标签、来源标识）可以适度格式化，但来源标识不得强制截断到极短长度。
- 内存中最多保留最近 300 条记录，超出时丢弃最旧记录；300 条限制不影响单条消息内容完整性。

## 1. 色彩系统

### 1.1 主色调 — 橙黄系

| 角色          | 色值 / Ink 颜色              | 用途                               |
|---------------|-----------------------------|------------------------------------|
| 品牌主色      | `#FF8C00` (dark orange)     | 全局顶栏边框、品牌标题、主要强调    |
| 高亮金色      | `#FFD700` (gold)            | 分区标题、选中行前缀光标 `❯`        |
| 活跃橙        | `#FFA500` (orange)          | 当前选中列表项文字                  |
| 浅橙背景感    | `#FFB347` (light orange)    | 可启动状态的服务横幅                |

> Ink 使用 chalk 渲染颜色，支持真彩 hex。实现时直接传 `color="#FF8C00"` 等 hex 字符串即可。

### 1.2 语义辅助色

| 语义         | 色值 / 别名           | 用途                                          |
|--------------|-----------------------|-----------------------------------------------|
| 成功 / 已连接 | `#52C41A` / `green`   | 渠道已连接状态、绑定成功提示                   |
| 警告 / 未就绪 | `#FAAD14` / `yellow`  | 渠道未配置、待生效绑定、不可启动状态            |
| 错误 / 失败   | `#FF4D4F` / `red`     | 渠道异常、操作失败、owner 冲突                 |
| 信息 / 收到   | `#69B1FF` / `blueBright` | 运行期入站消息日志行                        |
| 出站 / 发送   | `#95DE64` / `greenBright` | 运行期出站消息日志行                        |
| 进度 / 系统   | `#FFD666` / `yellowBright` | 运行期 Codex 进度日志行                    |
| 静音 / 次要   | `#888888` / `gray`    | 占位提示、次要说明、快捷键提示                 |
| 危险操作      | `#FF7875` / `redBright` | 删除渠道等破坏性操作高亮                     |

### 1.3 边框与分割线

| 元素              | 样式                                    |
|-------------------|-----------------------------------------|
| 全屏主框外边框    | 无单独框；用顶栏和底栏双横线 `═` 做区隔  |
| 信息面板内边框    | Ink `borderStyle="single"` + `#FF8C00`  |
| 分区标题下分割线  | `────────────────` 用 `gray` 渲染        |
| 顶栏 / 底栏分隔   | 用 `═══...═══` 全宽横线 `#FF8C00` 渲染   |
| 列表行活跃前缀    | `❯ ` (❯ + 空格)，`#FFD700` 金色          |
| 列表行非活跃前缀  | `  ` (两空格)                            |

## 2. 全局布局

### 2.1 全屏模式

Ink render 时传入 `{ alternateScreen: true }`（Ink v7 的全屏选项），并在 App 根节点使用 `process.stdout.columns` 和 `process.stdout.rows` 控制占满整个终端窗口。

```
┌ 全屏结构 ─────────────────────────────────────────── cols ──┐
│                                                              │
│  ▌ HEADER BAR ▐  (固定 1 行，品牌 + 全局状态)               │
│  ══════════════════════════════════════════════════════════  │
│                                                              │
│  ▌ CONTENT AREA ▐  (弹性高度，剩余 rows - 5 行)             │
│       各页面主内容在此区域渲染                                │
│                                                              │
│  ══════════════════════════════════════════════════════════  │
│  ▌ STATUS BAR ▐  (固定 2 行，flash 消息 + 快捷键提示)        │
│                                                              │
└──────────────────────────────────────────────────────────── ┘
```

说明：

- **HEADER BAR**：品牌名 `◈ CHAT CODEX` 左对齐用橙色，右侧显示全局状态（服务状态 + 权限模式），使用 `space-between` 排列，背景感靠下方双横线制造视觉分区。
- **CONTENT AREA**：各页面的主要信息区，高度为 `rows - 5`（顶栏 1 行 + 双分割线 2 行 + 底栏 2 行）。内容超出时从顶部开始自然滚动或做分区内滚动。
- **STATUS BAR**：第一行显示操作反馈（flash），第二行显示当前页快捷键提示。颜色按 flash 类型变色。

### 2.2 Frame 组件改造

现有 `Frame` 组件使用固定 `width={92}` + `borderStyle="round"` + `borderColor="cyan"`。

改造方向：

- 宽度改为 `width={Math.max(80, process.stdout.columns - 2)}`，贴满终端。
- 边框颜色改为 `#FF8C00`（橙色），保持 `borderStyle="round"` 但主框去掉圆角外框，改用上下分割线风格（见 2.1）。
- 标题渲染为 `◈ {title}` 加粗橙色 + 空格 + 副标题灰色，右对齐全局状态。

### 2.3 Section 组件改造

现有 `Section` 标题色为 `blue`。

改造方向：

- 标题改为 `#FFD700`（金色）加粗。
- 标题下方追加一条细分割线 `─────────...`（灰色），宽度填满面板内宽。
- 分区之间间距保持 `marginTop={1}`。

## 16. 响应式适配

### 16.1 宽终端（≥ 120 列）

- 渠道面板与配置面板使用左右两栏，`flexDirection="row"`。
- Session 选择页可在右侧追加 session 工作目录详情列。
- 运行期日志面板左右分栏（状态 40% + 日志 60%）。

### 16.2 标准终端（80–119 列）

- 默认单列布局，面板上下堆叠。
- 各面板宽度 `cols - 4`（预留左右各 2 格内外边距）。

### 16.3 窄终端（< 80 列）

- 只保留单列核心信息，长路径和长 ID 截断并追加 `…`。
- 操作区文字简短形式，例如「添加微信」代替「添加微信账号」。
- 快捷键提示栏允许换行。

### 16.4 高度自适应与视口滚动

#### 问题背景

`alternateScreen` 模式下，若页面内容行数超过终端高度，超出部分会被裁切，用户无法看到顶部或底部内容。Ink v7 导出了 `useWindowSize()` 钩子，可以实时获取 `{ columns, rows }` 并在终端 resize（SIGWINCH 信号）时自动触发重渲染。

#### 工具 API

```typescript
// ui-components.tsx 导出
export function useViewportRows(fixedRows: number): number
// 返回: Math.max(3, rows - fixedRows)
// fixedRows 是该 View 中除列表区以外的固定行数（Frame边框 + 各Section标题/分割线 + Footer）

export function visibleWindow<T>(items: T[], selected: number, maxVisible: number): {
  slice: T[];       // 当前可见切片
  startIndex: number; // 切片在原数组中的起始下标
  above: number;    // 切片之上还有多少条目
  below: number;    // 切片之下还有多少条目
}
// 以 selected 为中心，取 maxVisible 条，越界时对齐到边界

export function ScrollHint({ above, below }: { above: number; below: number }): JSX.Element
// 在列表上方显示 "↑ 还有 N 项"，下方显示 "↓ 还有 N 项"（THEME.muted 颜色）
```

#### 各 View 固定行数（fixedRows）参考值

| View | 固定行数 | 弹性区 |
|------|---------|--------|
| `ChannelsView`（有渠道） | 19 | 渠道列表（操作 7 项固定不滚动） |
| `BindingsView` | 8 | 绑定 + 待生效列表 |
| `SessionSelectView` | 10 | 可选 session 列表 |
| `WeixinBindingView` | 11 | 可选 session 列表 |
| `RuntimeLogView` | 22 | 日志条目区 |

> 固定行数 = Frame 边框(2) + 标题行(1) + 上边距(1) + 各 Section 标题+分割线(各 3 行) + 固定内容行 + Footer(2)。
> 数值为估算值，实现时以实测为准；窗口最小保证 3 行可见。

#### 渲染行为

1. `useViewportRows(N)` 在组件顶层调用，读取 `useWindowSize().rows`。
2. 将列表传入 `visibleWindow(items, selected, viewportRows)` 得到可见切片。
3. 在列表上方渲染 `ScrollHint above`，列表区渲染切片，列表下方渲染 `ScrollHint below`。
4. 切片中的索引 `startIndex + i` 与原始 `selected` 对比决定 `active` 状态，保持原有激活逻辑不变。
5. 终端 resize 时 `useWindowSize()` 触发重渲染，`viewportRows` 自动更新，列表无需手动刷新。

#### 不需要视口滚动的 View

- `HomeView`（最多 6 条操作，固定不可变）
- `ChannelDetailView`（最多 5 条操作）
- `PermissionView`（2 条选项）
- `WorkdirView`（2 条选项）
- 所有输入类 View（`AddFeishuView`、`ManualSessionView` 等）

## 17. 实施优先级与文件改动边界

### 17.1 仅修改展示层

本方案改动范围严格限制于：

| 文件 / 模块                        | 改动内容                                        |
|------------------------------------|-------------------------------------------------|
| `src/cli/tui/ui-components.tsx`    | 颜色常量、Frame / Section / ListRow / Footer 样式 |
| `src/cli/tui/views.tsx`            | 各 View 组件布局、面板边框、图标前缀             |
| `src/cli/tui/runtime-log.tsx`      | 日志行颜色、状态区布局、全屏适配                 |
| `src/cli/tui/run-tui.tsx`          | 添加 `alternateScreen: true` 渲染选项（Ink v7）  |
| `src/cli/tui/app.tsx`              | 顶栏面包屑逻辑（仅展示用）                       |

**不允许改动：**

- `src/cli/actions/` 目录（业务逻辑）
- `src/cli/tui/types.ts`（Screen 类型）
- `src/state/` 目录（状态存储）
- `src/bridge/` 目录（运行时核心）

### 17.2 颜色常量集中管理

建议在 `ui-components.tsx` 顶部或独立 `theme.ts` 中定义颜色常量：

```typescript
export const THEME = {
```

## 18. 验收标准

- 全屏时 Header + Content + StatusBar 各自占据正确区域，不截断不溢出。
- 橙色系主色在所有页面一致出现（边框、标题、活跃光标、品牌文字）。
- 成功 / 警告 / 错误三种语义色与橙色主色形成视觉对比，不混淆。
- 所有信息面板有明确边框分割，不同分区之间视觉层次清晰。
- 所有快捷键和按键逻辑与现有行为完全一致，不新增也不移除任何按键绑定。
- 宽终端（120+）和标准终端（80-119）均无文字重叠和字段错位。
- App Secret 不在任何页面回显。
- 运行期日志长文本不截断。
- `npm run chat-codex` 实际渲染效果与本文档线框图视觉一致。
